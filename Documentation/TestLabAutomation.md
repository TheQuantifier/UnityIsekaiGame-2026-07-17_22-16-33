# Automated Test Lab Scenario Runner

Maintenance M2 adds a Development-only automation runner for the existing Prototype Test Lab. It supplements Edit Mode tests, Unity Test Framework runs, definition validation, and manual exploratory testing. It is meant to verify prototype runtime composition: scene wiring, service instances, definition catalogs, event subscriptions, persistence registration, Actor/body resolution, and cross-feature runtime behavior.

## Architecture

The framework lives under `Assets/_Project/Development/TestLab/Automation/` and is compiled only for `UNITY_EDITOR || DEVELOPMENT_BUILD`. Runtime and Gameplay assemblies do not reference it. The permanent menu only sees the existing Test Lab menu extension; production builds can exclude Development without needing production code changes.

The core types are:

- `ITestLabAutomationSuite`: stable feature-level suite metadata and ordered scenarios.
- `ITestLabAutomationScenario`: stable scenario metadata and ordered steps.
- `ITestLabScenarioStep`: one action or assertion.
- `TestLabAutomationRunner`: selected scenario, suite, quick, all, rerun-failed, cancellation, and stop-on-first-failure execution.
- `TestLabAutomationRegistry`: deterministic Development-owned registration.
- `TestLabAutomationResult`, `TestLabScenarioResult`, and `TestLabAutomationStepResult`: immutable result snapshots.

Suites register through the Development Test Lab service. Adding a future suite should not require production runtime edits.

## Scenario Lifecycle

Each scenario runs on the Unity main thread:

1. Reset relevant Test Lab runtime state.
2. Run suite setup.
3. Run scenario setup.
4. Run ordered steps.
5. Stop the scenario on the first failed or errored step.
6. Run cleanup and teardown.
7. Reset after cleanup.
8. Return an immutable result.

Ordinary assertion failures are reported as failed step results. Unexpected exceptions are caught and reported as `Error`.

## Reset and Isolation

`PrototypeTestLabAutomationResetCoordinator` restores a known prototype baseline through existing Development adapters:

- Health, Mana, and Stamina are restored.
- temporary statuses are cleared.
- player and enemy defense state is cancelled.
- transient combat execution commitments are cleared.
- player and enemy combat state exits are requested.
- the prototype enemy is reset.
- event buffers are cleared.

The reset coordinator does not mutate canonical definition assets, authored stable IDs, project files, or editor preferences. Some systems with no complete public reset boundary are reset through the best existing Test Lab adapters and remain documented limitations.

## Assertions and Event Capture

`TestLabAssertions` supports reusable checks for equality, approximate equality, true/false, null/not-null, contains, count, sequence equality, expected success/failure, revision changed/unchanged, and validation result checks.

`TestLabAutomationEventCapture` records immutable summaries with event type, order, simulation time, transaction ID, source Actor, target Actor, and result code. It owns its subscriptions and removes them during disposal.

## Deterministic Time and Transactions

Automation transaction IDs are generated by `TestLabAutomationTransactionIds` from suite ID, scenario ID, run ID, step index, and operation name. Duplicate scenarios can deliberately reuse the same generated ID.

Feature-specific controllable clocks remain owned by existing Test Lab service methods such as combat state advancement, ongoing-effect ticking, and combat execution clock advancement. Automation calls those methods instead of waiting in real time.

## Run Modes

The UI supports:

- Run selected scenario.
- Run current suite.
- Run all quick scenarios.
- Run all registered scenarios.
- Rerun failed scenarios.
- Cancel remaining run.
- Stop on first failure.
- Clear results.

Cancellation is checked between scenarios and steps. The current synchronous step is allowed to finish before remaining scenarios are marked cancelled.

## Report Export

Reports are written outside `Assets` under `Temp/TestLabAutomation/`.

Supported formats:

- JSON for structured inspection.
- Markdown for readable review.

Reports include project name, Unity version, platform, run ID, run mode, start/end time, suite and scenario IDs, step statuses, expected/actual values, diagnostics, transaction IDs, exception details, and totals.

## Test Lab UI

The existing Test Lab manual controls remain available. The Automation section adds suite and scenario selectors, run buttons, rerun/cancel/clear controls, export buttons, validation, stop-on-first-failure, and auto-scroll toggles.

## Limitations

The runner does not replace Unity Test Framework, CI, screenshot comparison, input simulation, performance benchmarking, multiplayer orchestration, animation quality checks, or gameplay bots. It intentionally avoids scene traversal and broad physical input automation.
